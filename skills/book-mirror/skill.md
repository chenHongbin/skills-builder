# 书籍镜子 (Book Mirror)

> 受 Garry Tan 启发。把任何一本书变成一面镜子——左边是作者，右边是你自己。

## 概述

普通的 AI 书摘人人适用，也就人人没用。书籍镜子不同：它阅读 AI 对你的全部了解——你的项目、挑战、决策、价值观——然后告诉你**这些文字对你具体意味着什么**。

不只是书——任何文字都可以：一篇公众号文章、一段朋友圈、一份报告、一条推文。只要是你想深度理解的文字，书籍镜子都能照出它对你的独特意义。

**核心承诺：**
- 左栏忠实呈现作者——你可以跳过原书
- 右栏精准映射你——读起来像一个了解你的人在书页空白处给你写批注
- 如果某章与你无关，直说——不强行关联
- 不生成泛泛而谈的建议——每一条都指向你的具体情境

## 何时使用

### 触发条件
- 用户提供了一本书（文件路径、链接或粘贴文本）并希望获得个性化分析
- 用户提供了一段文字/公众号文章/朋友圈/社交媒体内容并希望个性化解读
- 用户说：分析这本书/这篇文章/这段文字、书镜、book mirror、帮我读、逐章解读、这段对我有什么用
- 用户想理解一段文字如何**对他/她具体**产生价值

### 何时不使用
- 用户只想做一个普通摘要（不想要个性化映射）——用普通对话即可
- 用户只是想问文中的某个具体问题——直接回答，不需要启动完整分析流程
- 用户提供的是图片、视频等非文字内容（无法提取文本）

## 工作流程总览

```
用户提供内容（书籍/文章/短文/链接）
    │
    ▼
阶段 0: 识别输入类型，判断内容长短
    │
    ├─ 短内容（< 10K 字符）─→ 阶段 3-短: 一次性两栏分析 → 完成
    │
    └─ 长内容（≥ 10K 字符）:
        │
        ▼
    阶段 1: 提取文本 → /tmp/book_mirror_full.txt
        │
        ▼
    阶段 2: 读取用户画像（从记忆中）
        │
        ▼
    阶段 3: 逐章分析循环 ← 本书的核心
        │  对每一批章节（1-3章）：
        │  1. 读取章节文本
        │  2. 输出两栏分析
        │  3. 询问是否继续
        │
        ▼
    阶段 4: 全书总结（可选，用户选择）
        - 跨章节主题串联
        - 按相关性排序的可执行要点
        - 值得回头重读的内容
```

## 阶段 0: 识别输入并收集信息

### 0.1 检测输入类型

根据用户提供的内容判断输入类型：

| 输入类型 | 检测信号 | 提取方式 |
|---------|---------|---------|
| PDF 文件 | 以 `.pdf` 结尾的路径 | `pdftotext -layout` |
| EPUB 文件 | 以 `.epub` 结尾的路径 | 内置 Python 脚本 |
| 公众号文章 | 含 `mp.weixin.qq.com` 的链接 | **优先使用 x-fetcher 或 x-reader skill**；不可用时降级为 mcp__fetch__fetch_url |
| X/Twitter 推文 | 含 `x.com` 或 `twitter.com` 的链接 | **优先使用 x-fetcher skill**；不可用时降级为 mcp__fetch__fetch_url |
| 小红书内容 | 含 `xiaohongshu.com` 的链接 | **优先使用 x-reader skill**；不可用时降级为 mcp__fetch__fetch_url |
| 普通网页链接 | 以 `http` 开头，非以上平台 | mcp__fetch__fetch_url（网页）或 curl + pdftotext（PDF 链接） |
| 直接粘贴的文本 | 用户直接在对话中粘贴 | 判断长度后保存到 `/tmp/book_mirror_full.txt` |
| 朋友圈/社交媒体文本 | 用户粘贴的短文本 | 直接保存到 `/tmp/book_mirror_full.txt` |
| Amazon/Goodreads 链接 | 含 "amazon" 或 "goodreads" | 无法抓取，告知用户并提供替代方案 |

### 0.2 如果用户没有提供内容

使用 AskUserQuestion 询问：

```
问题："你想分析什么内容？"
选项：
- "我提供文件路径（PDF/EPUB）"
- "我粘贴一段文字"
- "我提供链接（文章/公众号/网页）"
- "我提供 EPUB 文件路径"
```

### 0.3 收集阅读目标

在开始分析前询问用户（如果他们没有主动说明）：

> "为了给出更精准的分析：你读这段内容主要想解决什么问题？或者有什么想从中获得的？"

如果用户已经在对话中说明了阅读目的，直接记录，不再重复询问。

### 0.4 判断内容长度，选择分析模式

**这是关键的决策点。** 根据内容长度选择不同路径：

**判断方法：**
1. 如果内容已保存到 `/tmp/book_mirror_full.txt`，用 `wc -c` 检查字符数
2. 如果是用户刚粘贴的文本，根据粘贴内容的多少直接估算
3. 如果是 URL 提取的内容，根据提取结果判断

**路由规则：**
- **< 10,000 字符 → 短内容模式**：跳过章节检测，直接进入一次性两栏分析（见"短内容分析模式"）
- **≥ 10,000 字符 → 长内容模式**：继续阶段 1-4 的完整流程（提取→画像→逐章分析→总结）

**特殊情况：**
- 用户明确说"这是一本书/长篇报告"——即使字符数偏少也走长内容模式
- 用户明确说"帮我快速解读一下这段"——即使字符数偏多也走短内容模式
- 公众号文章通常 2K-8K 字符，自动走短内容模式
- 朋友圈/社交媒体通常 < 2K 字符，自动走短内容模式

向用户确认你的判断：
> "这段内容约 X,XXX 字，我用[短内容一次性分析/长内容逐章分析]模式来处理。"

## 阶段 1: 书籍文本提取

### 1.1 PDF 提取

使用 `pdftotext` 命令行工具（已验证安装在 `/usr/local/bin/pdftotext`）：

```bash
pdftotext -layout -enc UTF-8 "<用户提供的路径>" /tmp/book_mirror_full.txt
```

关键参数：
- `-layout`：保留物理布局，有助于章节标题检测
- `-enc UTF-8`：强制 UTF-8 编码，适用于中文内容

提取后操作：
1. 用 Read 工具检查输出文件的行数和字符数
2. 向用户汇报："已提取 XX 页，约 XX,XXX 字"
3. 如果文件超过 500K 字符，触发"超长书"边界情况处理

### 1.2 EPUB 提取

使用内置的 Python 脚本（纯标准库，不需要安装任何依赖）：

```bash
# 先用 ls 找到脚本的实际位置
python3 ~/.claude/skills/book-mirror/scripts/extract_epub.py "<用户提供的路径>" /tmp/book_mirror_full.txt
```

脚本输出 JSON 统计信息（提取的节数、跳过的节数、总字符数）。阅读这些统计信息并向用户汇报：
- "这本书共提取了 X 个章节片段，总计约 XX,XXX 字"
- 如果跳过很多节（例如照片页），简单说明：跳过了 Y 个图片/封面页

### 1.3 URL/链接提取

根据链接类型选择不同的提取方式。**关键原则：对于封闭平台（微信、X、小红书），优先使用专用 skill，不要直接用 mcp__fetch__fetch_url（通常会失败）。**

#### 1.3.1 公众号文章（`mp.weixin.qq.com`）

**方法 1（内置，零依赖，已验证有效）：curl + 微信 UA + 提取 js_content**

微信公众号的文章正文存储在 HTML 的 `<div id="js_content">` 中，只需用微信内置浏览器的 User-Agent 就能拿到。

```bash
curl -s -L \
  -H "User-Agent: Mozilla/5.0 (iPhone; CPU iPhone OS 16_0 like Mac OS X) AppleWebKit/605.1.15 (KHTML, like Gecko) Mobile/15E148 MicroMessenger/8.0.38(0x18002633) NetType/WIFI Language/zh_CN" \
  "<URL>" | python3 -c "
import sys, re, html
content = sys.stdin.read()
m = re.search(r'id=\"js_content\"[^>]*>(.*?)</div>\s*<script', content, re.DOTALL)
if m:
    text = re.sub(r'<[^>]+>', '\n', m.group(1))
    text = html.unescape(text)
    text = re.sub(r'\n\s*\n+', '\n\n', text)
    text = text.strip()
    if len(text) > 100:
        with open('/tmp/book_mirror_full.txt', 'w') as f:
            f.write(text)
        print(f'成功提取 {len(text)} 字符')
    else:
        print('提取内容过短，可能被反爬')
else:
    print('未找到文章正文，尝试方法 2')
"
```

如果提取成功（输出"成功提取 XX 字符"），直接进入阶段 0.4 判断长短内容模式。

**方法 2（备选）：使用 x-fetcher skill**
如果方法 1 失败，且用户已安装 x-fetcher（`~/.claude/skills/x-fetcher/`）：
```bash
python3 ~/.claude/skills/x-fetcher/fetch_wechat.py "<URL>" --save-md
```
然后将输出的 Markdown 文件内容保存到 `/tmp/book_mirror_full.txt`。

**方法 3（最后降级）：mcp__fetch__fetch_url**
如果方法 1 和方法 2 都失败，尝试 `mcp__fetch__fetch_url`（设置 `max_length=50000`）。通常只能拿到 JS 代码，拿不到正文。告知用户：
> "这篇文章自动提取失败了。你可以直接粘贴文章文字给我，我一样能分析。"

#### 1.3.2 X/Twitter 推文（`x.com` 或 `twitter.com`）

**方法 1（推荐）：使用 x-fetcher skill**
```bash
python3 ~/.claude/skills/x-fetcher/fetch_x.py "<URL>" --save-md
```
将输出的 Markdown 文件内容保存到 `/tmp/book_mirror_full.txt`。

如果用户想看完整讨论，可用 `--with-replies` 参数获取回复。

**方法 2（降级）：直接 fetch**
如果 x-fetcher 未安装，尝试 `mcp__fetch__fetch_url`，但告知用户可能失败。

#### 1.3.3 小红书（`xiaohongshu.com`）

**方法 1（推荐）：使用 x-reader skill**
如果用户已安装 x-reader，调用该 skill 读取内容。

**方法 2（降级）：直接 fetch**
如果 x-reader 未安装，尝试 `mcp__fetch__fetch_url`。小红书反爬严格，大概率失败。告知用户安装 x-reader：https://github.com/runesleo/x-reader

#### 1.3.4 PDF 链接（以 `.pdf` 结尾）
```bash
curl -L -o /tmp/book_mirror_download.pdf "<URL>"
pdftotext -layout -enc UTF-8 /tmp/book_mirror_download.pdf /tmp/book_mirror_full.txt
```

#### 1.3.5 普通网页链接（文章、博客等）
使用 `mcp__fetch__fetch_url`，设置 `max_length=50000` 获取文本内容。获取后保存到 `/tmp/book_mirror_full.txt`。

### 1.4 直接粘贴的文本

用户直接在对话中粘贴文本时：
1. 使用 Write 工具将文本保存到 `/tmp/book_mirror_full.txt`。
2. 用 `wc -c` 检查字符数。
3. 根据 0.4 节的规则判断：短内容→跳转到短内容分析模式；长内容→继续章节检测。
4. 不需要像以前那样"询问用户确认是否为摘录"——无论长短都分析，只是走不同路径。

### 1.5 文本清理

提取后统一执行。使用 Python 一行命令清理：
```bash
python3 -c "
import unicodedata, re
with open('/tmp/book_mirror_full.txt', 'r') as f:
    text = f.read()
text = unicodedata.normalize('NFC', text)
text = re.sub(r'\n{3,}', '\n\n', text)
text = ''.join(c for c in text if c.isprintable() or c == '\n')
with open('/tmp/book_mirror_full.txt', 'w') as f:
    f.write(text)
print(f'已清理，共 {len(text)} 字符')
"
```

### 1.6 推荐配套技能（可选，增强封闭平台支持）

**微信公众号已内置支持（见 1.3.1 方法 1），零依赖即可提取。** 以下技能用于增强其他平台的支持：

| 技能 | 解决什么问题 | 安装方式 | GitHub |
|------|------------|---------|--------|
| **x-fetcher** | 读取 X/Twitter 推文；微信公众号备选方案（方法 1 失败时） | `git clone` 到 `~/.claude/skills/x-fetcher/`，然后 `pip install -r requirements.txt` | https://github.com/Jane-xiaoer/x-fetcher |
| **x-reader** | 读取小红书内容 | `git clone` 到 `~/.claude/skills/x-reader/` | https://github.com/runesleo/x-reader |

**检测逻辑：**
```bash
ls ~/.claude/skills/x-fetcher/fetch_x.py 2>/dev/null && echo "x-fetcher 已安装" || echo "x-fetcher 未安装"
ls ~/.claude/skills/x-reader/ 2>/dev/null && echo "x-reader 已安装" || echo "x-reader 未安装"
```

**核心原则：能内置解决的就不依赖外部 skill。微信公众号用内置 curl 方案（1.3.1 方法 1），失败才降级到 x-fetcher。X/Twitter 和小红书优先用专用 skill，没有则尝试 mcp__fetch__fetch_url，再失败就请用户粘贴文字。**

## 阶段 2: 读取用户画像

### 2.1 从以下来源收集信息（按优先级）

**来源 1：MCP 记忆知识图谱**
如果 `mcp__memory__read_graph` 和 `mcp__memory__search_nodes` 工具可用，使用它们读取记忆中关于用户的信息。重点寻找：
- 身份角色、行业、职业阶段
- 当前项目和目标
- 核心挑战与痛点
- 关系和团队结构
- 过去的决策及其背景
- 价值观和偏好

**来源 2：全局 MEMORY.md**
读取 `~/.claude/memory/MEMORY.md`。这里通常包含用户显式设定的偏好、项目联系人、决策风格等。

**来源 3：项目级 CLAUDE.md**
读取当前工作目录的 `./CLAUDE.md` 或 `CLAUDE.md`。包含项目级上下文。

**来源 4：对话自我反思**
从当前对话中提取你已经了解到的用户信息：沟通风格、当前关注的话题、情绪状态。这些隐式信息往往最相关。

**来源 5：显式询问（备选）**
如果以上来源都没有足够信息，直接问用户 2-3 个定向问题：
- "为了让分析真正触及你：你目前的工作/角色是什么？"
- "你当前面临的最大挑战是什么？"
- "你读这本书最想解决什么问题？"

### 2.2 构建画像（内部使用，不向用户展示）

将所有来源综合成一个结构化的用户画像。这个画像不在对话中展示，而是作为右栏分析的依据：

```
用户画像 (internal):
- 身份角色: [行业、角色、阶段]
- 当前项目: [正在做的事]
- 核心挑战: [1-3 个痛点]
- 决策偏好: [做决策的方式]
- 价值观: [看重什么]
- 知识背景: [已知/未知的领域]
- 关系网络: [团队、利益相关者]
- 近期关注: [最近在想什么]
```

### 2.3 记忆稀疏时的处理

如果用户信息很少（新用户）：
1. 不要编造画像。
2. 如实告知："我目前对你的了解有限，所以某些部分的分析可能不够深入。随着我们对话增多，它会越来越精准。"
3. 基于通用人类模式做最佳推测（每个人都有目标、挑战、关系），但要标明"这是我的推测"。
4. 仍然尽力——即使是部分个性化的分析也比通用摘要强。

## 阶段 3: 逐章分析循环（长内容模式）

> 此阶段仅用于长内容（≥ 10K 字符）。短内容请跳转到"短内容分析模式"。

### 3.1 章节检测

使用内置的 Python 脚本检测章节边界：

```bash
python3 ~/.claude/skills/book-mirror/scripts/detect_chapters.py /tmp/book_mirror_full.txt
```

脚本输出 JSON 格式的章节列表。解析输出，向用户展示章节概览。

### 3.2 处理大量章节

如果检测到超过 30 个章节：
- 告知用户："这本书检测到了 X 个章节，全部逐章分析需要很多轮对话。"
- 提供选项：
  - "全部分析（可能需要 X 轮对话）"
  - "只看前 5 章"
  - "只看关键章节（你可以指定）"
  - "让我合并相关章节，压缩到 10-15 组"

### 3.3 章节批处理节奏

根据每章字符数自动决定每轮处理的章节数：
- 短章（< 5K 字符）：每轮处理 3-4 章
- 中等篇幅（5K-10K 字符）：每轮处理 2 章
- 长章（> 10K 字符）：每轮处理 1 章

在第一轮分析前告诉用户计划：
> "共 X 章。我计划分成 Y 轮来分析，每轮 1-3 章。准备好了吗？现在开始第一章。"

### 3.4 阅读章节文本

对当前批次的每个章节，使用 Read 工具读取：
```
Read /tmp/book_mirror_full.txt offset=<start_char> limit=<char_count>
```

### 3.5 Token 管理

**关键约束：不要把整本书一次性装入上下文。**
- 每轮只加载当前批次的章节文本
- 用户画像保持在 1K tokens 以内
- 前几章的 1 行小结用于上下文连贯性
- 总 token 预算：每轮 5K-15K tokens

**超长书协议**（全书 > 500K 字符）：
1. 警告用户这本书非常长
2. 提供选择：
   - 只看特定章节
   - 略读模式：只看每章开头和结尾
   - 概述模式：检测关键主题，代表性分析
   - 仍然全部分析（用户接受长流程）

**进度保存：** 每轮结束后，保存进度到 `/tmp/book_mirror_progress.md`：
```markdown
# Book Mirror 进度
- 书籍：[书名]
- 已完成章节：第1-X章
- 关键主题识别：[主题列表]
- 用户画像摘要：[简要]
```

### 3.6 输出：两栏分析模板

对每个章节，按以下格式输出：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📖 第X章：[章节标题]
   (约 Y,YYY 字 · 检测方法: [方式])
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

本章核心：[1-2 句话概括作者在本章的核心论点和关键框架]

| 作者说了什么 | 这对你意味着什么 |
|:------------|:----------------|
| [作者的核心论点1——忠实复现，不是你的总结] | [基于你的具体情境的应用——用你的名字、你的项目、你的处境] |
| [作者的核心论点2] | [个性化的含义或行动建议] |
| [作者引入的关键概念/框架] | [这个概念如何连接到你当前在做的具体事情] |
| [作者的建议或处方] | [你是否应该采纳，如何采纳，具体的下一步] |
| [值得记住的一句话/引用] | [这句话为什么对你来说不一样——基于你是谁] |

💡 本章对你最关键的一句话：[针对你的情况，1 句话总结最有价值的收获]
```

### 3.7 左栏规则

左栏的任务是忠实呈现作者——不是你对作者的理解，而是作者本身：
- 捕捉作者的**实际论点**，不是你的概括
- 保留作者的**语气和框架**
- 包含作者使用的**标志性短语或框架名**
- **不要**添加你的评价——那是右栏的事
- 如果作者某处有错或过时了，在**右栏**指出，不在左栏

### 3.8 右栏规则（核心！）

右栏是镜子——它的质量决定这个 skill 的价值。

**必须做到 ✅：**
- 引用用户的实际上下文（名字、角色、项目、情境）
- 具体化："因为你正在做 [X]，这个框架可以帮你 [Y]"
- 恰当挑战："这跟你目前在 [Z] 上的方法有矛盾——值得琢磨一下"
- 给出具体下一步："试试这样做：[具体行动]"
- 连接到用户说过或做过的事
- 区分"这直接适用"和"这有趣但跟你关系不大"

**绝对不能 ❌：**
- 泛泛的自助建议（"你可以反思一下这个"）
- 无个性的观察（"这对科技行业的任何人都有参考价值"）
- 模糊的应用（"这对你的职业发展可能有帮助"）
- 用不同的话复述左栏内容
- 编造你并不了解的关于用户的事
- 奉承（"你已经做得很好了！"——没有增量信息）

**特异性测试：** 把你写的右栏内容中的用户相关词替换成 "[某人]"。如果替换后读起来仍然像有效的建议——那太泛了。好的右栏在去个性化后会变得毫无意义。

**差（❌）的例子：**
> "这个框架可以帮助你更系统地思考战略问题。"
> → 谁都可以用这句话。差评。

**好（✅）的例子：**
> "你最近在决定练吧是要做功能 X 还是 Y，这个框架的建议是：优先做那个能降低用户入门焦虑的功能——因为你之前提到过，你的用户最核心的需求不是更多功能，而是信心。"
> → 具体到了项目名、具体困境、具体用户需求。好评。

### 3.9 如果某章不适用

不要强行关联。如果某一章跟用户的情况真的没有交集：
- 在右栏写："本章主要集中在 [某话题]，与你的当前情况关联不强。如果你未来进入 [某场景]，可以回头来看。"
- 如果有相关联的**部分**，提取那一部分进行分析
- 短的坦白比长的编造强一百倍

### 3.10 质量自检清单

**每章呈现前，在心里过一遍：**
- [ ] 左栏：忠实呈现了作者的论点（不是我的概括）
- [ ] 右栏：引用了至少 2 个用户画像中的具体信息
- [ ] 右栏：包含至少 1 个具体行动或问题
- [ ] 右栏：不包含任何泛泛而谈的建议
- [ ] "最关键的一句话"对这个用户是唯一的，不是书评套话
- [ ] 章节核心概括是准确的（不是编造的）

### 3.11 用户反馈循环

每 3-4 章后，问用户：
> "分析到现在感觉怎么样？太泛了？太细了？还是有什么想调整的方向？我可以调整分析的角度。"

## 短内容分析模式

> 适用于：公众号文章、朋友圈、推文、短文、报告摘要、任何 < 10K 字符的文字内容。

### 工作流程

```
阶段 0: 识别为短内容
    │
    ▼
阶段 2: 读取用户画像（同长内容模式，共享 2.1-2.3 所有规则）
    │
    ▼
一次性分析:
  1. 用 Read 工具读入全文（短内容可以直接全量加载）
  2. 识别内容类型和核心主题
  3. 输出两栏分析（格式见下方模板）
  4. 完成——不需要逐章循环
```

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

核心主旨：[1-2 句话概括这段文字的核心观点和背景意图]

|:------------|:----------------|
| [作者的核心观点1——忠实复现] | [基于你的具体情境的应用] |
| [作者的核心观点2] | [个性化的含义或行动建议] |

### 短内容分析的特殊规则

**相比长内容的不同之处：**
- 不需要章节检测——全文即一个分析单元
- 内容类型标记（公众号文章/朋友圈/推文/短文）帮助用户定位
- "延伸思考"行可选——如果用户的画像中有明显关联的信息才加
- 一次性输出，不需要询问"是否继续"

**与长内容相同的部分（严格遵守）：**
- 左栏规则（3.7）、右栏规则（3.8）完全适用
- 质量自检清单（3.10）完全适用
- 如果内容不适用（3.9）完全适用
- 常见反模式完全适用
- 右栏特异性测试完全适用

### 公众号文章特别处理

微信公众号文章（链接含 `mp.weixin.qq.com`）：
1. **内容提取**：优先使用 x-fetcher 或 x-reader skill（见 1.3.1 节），直接 fetch 通常会失败
2. 公众号文章通常有标题、作者、发布时间——提取这些元信息
3. 可能有"推荐阅读""广告"等尾随内容——分析时聚焦正文
4. 文章末尾的"在看""点赞"等互动提示可以忽略
5. 如果提取的内容含大量 HTML 标签残余，用文本清理（1.5 节）处理

### 朋友圈/社交媒体特别处理

朋友圈或社交媒体内容通常极短（< 500 字符）：
1. 简洁即可——不需要凑 5 行两栏分析
2. 重点放在"这句话为什么是这个人在这个时刻说的"以及"它触动你的点在哪"
3. 如果内容太短（< 100 字符），两栏可以缩减到 2-3 行核心要点

**内容**: 一条朋友圈：

## 阶段 4: 全书总结（可选，仅长内容）

所有章节完成后，询问用户是否需要全书总结。如果用户同意，生成：

### 4.1 跨章节主题串联
找出横跨多个章节出现在用户镜子中的主题（例如：某个模式反复出现，某条线索逐步深化）。

### 4.2 可执行要点（按相关性排序）
不列书评式的"你该从这本书学到的 3 件事"，而是：
> "基于你当前的 [具体情境]，以下是本书对你最有杠杆效应的 5 个洞察："

每个要点必须：
- 引用书中的具体内容（哪章、作者怎么说）
- 解释为什么**对你**重要（用画像中的信息）
- 给出一个可操作的下一步

### 4.3 值得回头重读的章节
哪些章节在当前阶段对你参考意义不大，但未来（例如当你进入某个阶段、面对某个决策时）值得重新翻阅。

## 边界情况

### 1. 超长书（> 500K 字符）
- 立即警告用户并给出缩减选项（见 3.5 节超长书协议）
- 千万不要静默加载整本书导致 token 溢出

### 2. 无法检测到章节结构
- 告知用户："这本书的章节标记不够清晰。"
- 提供备选方案：
  - 按约 15K 字符自动分段
  - 用户手动指定分段点
  - 对短内容（< 30K 字符），当作一个"章节"处理

### 3. 用户记忆稀疏
- 按照 2.3 节的策略处理
- 不编造、不自卑——即使是部分个性化的分析也有价值
- 特别标记推测性的内容："[推测]：如果你的情况是这样，那么..."

### 4. 非书籍内容（文章、论文、报告）
- 自动走短内容模式或长内容模式（取决于字符数）
- 对短文（< 10K 字符），直接一次性分析，不需要分章节
- 对长篇论文和报告，按"Abstract/Introduction/Methods/Results/Conclusion"结构检测章节
- 所有内容类型共享同一套两栏分析质量规则

### 5. 用户同时提供多本书
- 一次只能分析一本书。请用户选择先分析哪本。

### 6. 外文书籍
- 如果检测到主要语言不是中英文，告知用户
- 仍然提取和检测章节，但在分析时标注原语言
- 关键引用保留原文加翻译

## 常见反模式（不要做的事）

| 反模式 | 为什么失败 | 修复方法 |
|--------|-----------|---------|
| **写读后感** | 右栏只是在用不同的话复述左栏 | 右栏必须引用用户上下文 |
| **写星座运势** | 模糊的个人化表述适用于任何人（"你正处于人生的转折期..."） | 每条声明必须引用用户记忆中的具体证据 |
| **写啦啦队** | "你已经在做这个了！很棒！"——零增量信息 | 如果用户确实在做，说清楚如何做得更好，或作者会批评什么 |
| **写第二堂课** | 右栏变成另一段讲解，而不是映射用户 | 右栏聚焦应用，不是解释 |
| **写幽灵** | 右栏只有一两个词（"Interesting." "Relevant."） | 每个左栏的观点都应有右栏对应 |
| **编故事** | 编造不在记忆中的用户事实 | 只用已知的信息；不确定时标注："如果你的情况是 [X]，那么..." |

💡 本章对你最关键的一句话：练吧现在不是在和竞争对手打仗，而是在和"用户觉得没练也行"这个惯性打仗。Mo(o)re 的建议是：不要做更通用的平台，做一个对某群人不可替代的工具。

## 禁止事项

**绝对不要做的事：**
- ❌ 一次性把整本书加载到上下文中（会导致 token 溢出）
- ❌ 在没有用户画像的情况下生成右栏分析（等于生成普通书摘）
- ❌ 编造或猜测用户的个人信息
- ❌ 用 AI 书评的方式总结书的内容（左栏是呈现作者，不是概括作者）
- ❌ 对不相关的章节强行建立联系
- ❌ 在长内容模式下，在没有让用户确认章节列表前就开始逐章分析
- ❌ 在短内容模式下，强行把短文拆成"假章节"来逐章分析

**每次对话开始时的验证：**
- [ ] 确认用户提供了内容（或询问用户提供）
- [ ] 确认输入类型并正确路由到提取方法
- [ ] 判断内容长度，选择正确的分析模式（短内容 vs 长内容）
- [ ] 确认记忆系统可用（或在缺失时告知用户）
- [ ] 长内容模式：展示章节列表并获得用户确认后才开始分析
- [ ] 每段/每章都执行右栏特异性测试
